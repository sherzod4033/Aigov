# План улучшения Retrieval (Поиска) в RAG-системе

## Цель
Устранить проблемы с потерей контекста при сложных многосоставных запросах (Multi-hop) и повысить точность поиска (особенно для таджикского языка) с помощью архитектуры Retrieve & Re-rank.

## Шаги реализации

### 1. Увеличение первичной выборки (High Recall)
* Файл: `soliqai/backend/app/modules/chat/service.py` и `ask/service.py`
* Логика: Изменить параметр `search_limit` или `n_results` при запросе к ChromaDB.
* Было: Запрашивалось небольшое количество чанков (`top_k`, обычно 4-5).
* Станет: Всегда запрашивать `top_k_retrieval = 15-25` кандидатов на первом этапе.

### 2. Добавление Reranker (Cross-Encoder)
* Интегрировать легковесную модель для переранжирования (например, `FlashRank`, `BAAI/bge-reranker-base` через HuggingFace/ONNX, или специализированный API).
* Логика: Пропускать найденные 15-25 чанков через Reranker вместе с исходным запросом пользователя.
* Reranker оценит семантическую связь (relevance score) каждой пары "запрос-документ".

### 3. Фильтрация итогового контекста (High Precision)
* Логика: После работы Reranker-а отсортировать результаты по убыванию relevance score.
* Оставить только `final_context = 4-6` самых релевантных чанков.
* Передавать в генератор (LLM) только этот очищенный топ-контекст.

### 4. Создание инструмента тестирования (Retrieval Eval)
* Создать отдельный скрипт: `soliqai/backend/scripts/evaluate_retrieval.py` (или добавить тесты в `tests/test_retrieval.py`).
* Назначение: Скрипт должен принимать набор вопросов (на русском и таджикском) и проверять, попадают ли нужные ID чанков в топ-5 выдачи.
* Генерация LLM (LLM Generation) в этом тесте должна быть отключена, чтобы изолированно замерять только метрики поиска (Recall@5, MRR).

## Исполнение
После утверждения данного плана, задача будет делегирована агенту для внесения изменений в код.
